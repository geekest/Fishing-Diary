# AGENTS.md

本文件是仓库级 AI Agent 协作规则，适用于本仓库全部目录。Agent 在修改任何文件前必须先阅读并遵守本文件；如子目录存在更近的 `AGENTS.md` 或 `AGENTS.override.md`，以更近规则为准。

## 项目定位

- 当前项目：`Fishing Diary`（钓鱼日记），面向钓友的 iOS MVP 原型，用于记录出钓、整理渔获照片，并生成适合社交平台发布的渔获分享卡。
- 技术栈：Swift 5.9、SwiftUI、SwiftData、XcodeGen、TOCropViewController，目标平台为 iOS。`project.yml` 当前配置最低 iOS 18.0，`README.md` 中的 iOS 16.0+ 描述需要后续确认是否同步。
- 核心模块：
  - `FishingDiary/App/`：App 入口与根视图。
  - `FishingDiary/Common/`：主题、通用组件和卡片视图。
  - `FishingDiary/Models/`：SwiftData 模型与记录流程状态。
  - `FishingDiary/Resources/`：Info.plist、Assets、Entitlements。
  - `FishingDiary/Services/`：天气、购买、图片渲染服务。
  - `FishingDiary/Views/`：引导、记录、日记、分享、我的页面。

## 开发原则

1. 修改前先理解现有架构、调用链、相邻代码和文档约定。
2. 优先复用已有代码、组件、模型和服务，不凭空新增平行实现。
3. 避免无必要重构，不为当前需求之外的问题扩大范围。
4. 修改范围保持最小，优先提交可审查、可验证的小步改动。
5. 不确定需求会影响实现方向时，先提出最多 3 个澄清问题；不阻塞实现的不确定性可以基于现有代码做合理假设，并在交付时说明。
6. 不修改业务代码以外的内容，除非任务明确需要或验证必须。

## 代码规范

- 保持已有 Swift、SwiftUI 和项目目录风格。
- 函数、变量、类型、文件命名使用英文，面向用户展示的前端文案可以使用中文。
- 新增代码必须考虑可维护性、可读性和边界场景。
- 避免重复逻辑；已有组件、主题、模型或服务可满足需求时应优先复用。
- 对复杂逻辑、关键分支、边界处理和非显然决策添加必要中文注释；简单代码不要添加噪音注释。
- 文档、注释和用户可见文案中，中文与英文单词、数字、行内代码之间保留一个半角空格。
- 不要在 import 外层添加 try/catch 或等价包装。

## 安装、启动与构建

- 工程由 XcodeGen 维护；修改 `project.yml` 后运行：`xcodegen generate`。
- 本地打开工程：`open FishingDiary.xcodeproj`。
- 可用构建命令待确认；在 macOS + Xcode 环境中优先使用 `xcodebuild` 针对可用模拟器验证。
- 当前仓库未发现自动化测试目录、CI workflow 或明确的 lint/typecheck 脚本；如新增测试或脚本，应同步更新本文件。

## 测试与验证

- 修改 Swift 代码后，优先运行可用的 Xcode build/test 命令。
- 修改 `project.yml` 后，必须重新生成 Xcode 工程并检查 diff 是否符合预期。
- 修改 UI 或可运行 Web/App 可视内容时，如环境允许，应提供截图或说明无法截图的环境限制。
- 若验证因缺少 Xcode、模拟器、依赖或网络等环境限制无法执行，必须在最终回复中明确说明。

## Git 工作流

- 开始修改前先查看当前分支和工作区状态。
- 不直接操作主分支；如当前在主分支，应先停下并说明风险。
- 不覆盖用户已有未提交改动；如发现非本次产生的改动，先说明再继续。
- 所有 commit message、Pull Request title、Pull Request description 默认使用中文；除非用户明确要求英文，否则禁止自动生成英文 PR 文档。
- commit 标题格式：`[类型] 简洁描述`，例如：`[功能] 新增用户登录模块`、`[修复] 更新枚举值`、`[优化] 提升代码可靠性并修复 bug`。
- commit 描述不超过 30 个中文汉字，需说明做了什么。
- PR 标题格式：`简洁描述`，例如：`新增用户登录模块`。
- PR 描述必须包含变更原因、实现方案，并采用以下中文结构：

```markdown
# 背景

说明为什么需要这个修改。

# 修改内容

列出主要变化。

# 测试验证

说明如何验证。

# 风险说明

说明潜在影响。
```

- 禁止使用 `Motivation`、`Description`、`Testing` 等英文 PR 模板。
- 分支命名遵循当前任务上层规则；如上层规则冲突，以用户本次明确要求和当前仓库托管约束为准，避免擅自重命名分支。

## 依赖与高风险变更

- 新增生产依赖前必须说明必要性、替代方案、影响范围和回滚方式，并等待确认。
- 修改数据库 schema、SwiftData 模型持久化语义、migration、权限、鉴权、支付、生产配置、entitlements、环境变量、CI/CD、部署脚本前，必须先说明风险并等待确认。
- WeatherKit、StoreKit 2、CloudKit/iCloud 当前为 Mock 或预留能力；接入真实服务前必须单独评估签名、权限、配置和上架影响。

## 完成标准

- 总结本次修改涉及的文件和行为变化。
- 运行与改动相关的测试、构建、lint、format 或 typecheck；无法运行时说明原因。
- 自行审查 diff，确认未引入无关重构、重复逻辑、不安全实现或边界问题。
- 明确说明尚未验证的内容和潜在风险。

## Agent 自我修正机制

如果 Agent 在任务中出现以下错误：

- 忽略已有规则。
- 修改范围过大。
- 生成不符合项目规范的文件。
- 重复犯同类错误。

必须在本文件中追加新的规则，使用以下格式：

```markdown
## Learned Rules

- 日期：YYYY-MM-DD
- 问题：发生了什么偏差。
- 新增约束：后续必须遵守的具体规则。
```
